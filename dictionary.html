Competition Rules

Everything you need to know about scoring and format

Scoring formula How leaderboard positions convert to competition points
critical
▶
Score = (n − 1)² where n = leaderboard position that round
Rank 1 (first)Score = 0 — best possible
Rank 4Score = 9
Rank 7Score = 36
Rank 42Score = 1681 — catastrophic

The quadratic penalty means one rank-42 round (1681 points) costs more than 46 rank-4 rounds (9 each = 414 total). Avoiding catastrophic rounds matters more than winning rounds.

Scores are summed across all rounds. Lower aggregate score = better final rank. Consistent top-5 finishes beat volatile performance.

Round structure 15-minute rounds, 10 days each, increasing difficulty
timing
▶
Round duration15 minutes = 900 seconds
Days per round10 days × 90 seconds = 900 ticks-time
Ticks per second5
Ticks per day450 (5 ticks/sec × 90 sec)
Earnings timingSecond 22 and second 88 of every day → 20 events/round
Earnings in ticksTick 110 and tick 440 within each day
Difficulty scalingLater rounds: opponent bots faster, spreads tighter, assets more volatile
PositionsReset to zero at start of each round
Auto-flattenBot flattens all positions at T-20s before round end
Position limits (exchange) Hard limits enforced by the exchange — violations → order rejection
hard limit
▶
Max order size40 units per single order
Max outstanding vol120 units total unfilled orders per symbol
Max open orders50 resting limit orders per symbol
Max absolute position200 units (long or short) per symbol

Exceeding any of these → order rejected silently. The bot does not always know which limit it hit. Our safe_place_order checks all four before submitting.

Stock B Semiconductor company. No direct news. Price known only via options PCP.
no direct newsimplied via options
▶

There is no earnings or news for B. Its true price is inferred using Put-Call Parity across all three option strikes.

B_implied = C_mid − P_mid + K (per strike, then weighted average)

Weights = 1 / (C_spread + P_spread). Tighter-spread strikes → more reliable estimate → higher weight.

B_implied is used by: (1) ETF NAV calculation, (2) optional PCP arb check for options strategy.

A

Stock A Strategy

StockAStrategy class — primary alpha engine

Earnings sweep Unconditional book sweep immediately after every EPS release
highest alpha
▶

When EPS fires: new FV = EPS × PE_A × EPS_SCALE_A. All asks below FV−2 are immediately swept (BUY). All bids above FV+2 are immediately swept (SELL). Unconditional means we do this regardless of current position — we sweep to the 200-unit limit.

sweep edge = 2 # ticks of buffer buy if ask_price < FV − 2 → guaranteed profit = FV − ask_price − 2 sell if bid_price > FV + 2 → guaranteed profit = bid_price − FV − 2

Post-earnings aggressive mode lasts 2 seconds (POST_EARNINGS_SECS = 2.0). During this window, on_book_update also calls _sweep_book continuously to catch any new stale orders that appear.

EPS_SCALE_A auto-calibration Self-corrects if the initial scale assumption is wrong
safety
▶

On every earnings event, the bot records the implied scale: implied = book_mid / (eps × PE_A). After 3+ observations, it takes the median (not mean — resistant to a single bad reading) and updates EPS_SCALE_A if it differs by more than 1.0.

# Example: eps=1.126, PE_A=10, book mid=1126 implied_scale = 1126 / (1.126 × 10) = 100.0 # After 3 obs with implied≈100: EPS_SCALE_A = 100

Persistence: _scale_obs list is NOT cleared on round reset. Calibration accumulates across the entire competition session. By round 3, scale is rock-solid before the first earnings fires.

Sanity bounds: only observations where 20 < implied_scale < 500 are recorded. Extreme values are discarded as likely noise from a thin book.

Pre-cancellation timer Cancels all A quotes 3 seconds before expected earnings
timing edge
▶

After each earnings event, EarningsTimer calculates exactly when the next earnings will fire (either 66 seconds or 24 seconds away, depending on which of the two daily events just fired). It sets an alarm for 3 seconds before that time.

Why: In harder rounds, faster bots have already priced in the next EPS before the news event fires. If our quotes are resting at stale prices in those final 3 seconds, they get picked off. Pre-cancel avoids this adverse selection.

After second 22 firesNext earnings in 66 seconds (88 − 22 = 66). Cancel at T+63s.
After second 88 firesNext earnings in 24 seconds (90 − 88 + 22 = 24). Cancel at T+21s.
PRE_CANCEL_SECS3.0 seconds
Passive market making Quotes bid/ask around FV with inventory skew and flow adjustment
spread income
▶
bid = FV_adj − spread/2 − total_skew
ask = FV_adj + spread/2 − total_skew
BASE_SPREAD4 ticks half-spread at zero inventory
Dynamic spreadWidens as position grows: spread = 4 + int(|pos|/100 × 4)
Inventory skew0.15 × pos — shifts both quotes away from heavy side
Flow skewFlowTracker.fv_adjustment() × 0.5 — adjusts toward smart money direction
At pos = +200bid = 0 (suppressed). Ask continues normally to reduce long.
At pos = −200ask = 0 (suppressed). Bid continues normally to reduce short.
Re-quote rate limitMaximum once per second (avoids cancelling/replacing every 100ms)
Quote size10 at normal inventory; max(2, 10//3)=3 near soft limit (100)

Cross-book check: if bid ≥ best_ask, set bid=0. If ask ≤ best_bid, set ask=0. This check is unconditional (applies even at position limits). Old bug: check was gated on abs(pos) < MAX_ABS_POS, allowing spread-crossing at limits.

FlowTracker (ALPHA = 0.05) Detects smart money direction from aggressive trade prints
signal
▶

Observes every trade print. If trade price ≥ best_ask → aggressive buy → +qty to rolling window. If price ≤ best_bid → aggressive sell → −qty. Ambiguous trades (between spread) are ignored.

fv_adjustment = ALPHA × net_signed_volume
ALPHA = 0.05, window = 60 recent trades

Applied to FV for both quoting and snipe decisions. Positive net signed volume → FV shifts up (smart money is buying). Negative → FV shifts down.

Safe to use because EPS_SCALE_A is now auto-calibrated. Without correct scale, a wrong FV could create a death spiral (wrong FV → sell → flow sees sell pressure → FV drops further → more selling). With correct scale, flow is a genuine edge.

Stock C Strategy

StockCStrategy class — ERC empirical model

Three-phase lifecycle C trades in three distinct phases per round — only Phase 3 has directional alpha
important
▶
Phase 1Before first C earnings. FV=None. Passive MM ±12 ticks, 3 units/side. No sniping. No directional trades.
Phase 2After first C earnings, before first calibration CPI. PE_C_scale known. FV still None (no ERC anchor yet). Wide passive MM continues.
Phase 3After first significant CPI. Sensitivity calibrated. FV active. Tight MM ±5 ticks / 8 units. Sniping enabled. Directional CPI trades.

The transition to Phase 3 requires: |ΔERC| > 2.0 from the CPI print. A CPI that barely moves probabilities (surprise near zero) does not calibrate.

pre_cpi_snapshot() Must be called BEFORE Fed processes CPI — captures baseline state
order-critical
▶

Captures: current ERC (_pre_cpi_erc) and current C book mid (_pre_cpi_mid).

This snapshot is essential for two purposes:

  1. Calibration: ΔERC = new_ERC − pre_cpi_ERC and ΔC = new_C_mid − pre_cpi_mid give the empirical sensitivity.
  2. Directional trades: expected_dc = sensitivity × ΔERC. We need ΔERC relative to the pre-CPI baseline.

If pre_cpi_snapshot is called AFTER Fed updates probabilities, both measurements will be post-CPI and ΔERC will be 0. The entire C model depends on this call order being correct.

Call order in bot_handle_news (CPI):

1. strat_c.pre_cpi_snapshot() # snapshot BEFORE ERC changes 2. strat_fed.on_cpi(fore, act) # Fed updates probabilities → ERC changes 3. strat_c.recalc_fv() # update C FV with new ERC 4. await strat_c.on_cpi(fore, act) # calibrate / directional trade 5. await strat_fed.trade() # immediate Fed trade on news
Sensitivity calibration Single-parameter fit from first significant CPI. No gradient descent.
replaces old BETA/GAMMA
▶
sensitivity = ΔC_mid / ΔERC
where Δ = post-CPI minus pre-CPI snapshot

This works because the full C formula is approximately linear in ERC for small deviations. The gradient dC/dERC ≈ −β × (EPS×PE0×γ + λ×B0/N×D) is a single round-constant that we measure directly.

Guard conditions for calibration:

  • |ΔERC| must exceed 2.0 (too small = noise-dominated)
  • If |ΔC_mid| < 1.0 but |ΔERC| > 3.0: wait for a cleaner observation (C book may not have repriced yet)
  • First significant CPI calibrates and does NOT trade. All subsequent CPIs trade directionally.

Anchor update on C earnings: When EPS changes, we adjust anchor_mid by (eps_new − eps_old) × PE_C_scale so the anchor remains valid after the earnings shift. PE_C_scale persists across rounds; sensitivity resets each round.

CPI directional trade Trade C before the market reprices after each CPI event
new alpha
▶

On each CPI event (after calibration), if the expected C price change exceeds the minimum threshold:

expected_dc = sensitivity × ΔERC if expected_dc < -4: # C should fall — sell before market reprices qty = min(10, C_MAX_POS + pos) # up to 10 units, never exceed ±20 SELL at best bid elif expected_dc > +4: # C should rise — buy before market reprices qty = min(10, C_MAX_POS - pos) BUY at best ask
CPI_TRADE_QTY10 units per event
C_MAX_POS±20 units from CPI directional trades alone
CPI_MIN_EXPECTED_DC4 ticks — minimum change to bother trading
max_abs_position["C"]50 (enforced by safe_place_order)

Half-Kelly sizing Position size proportional to edge, capped at MAX_POS=25
sizing
▶
For BUY: kf = (our_p − mkt_p) / (1 − mkt_p)
For SELL: kf = (mkt_p − our_p) / mkt_p
size = max(0, min(floor(0.5 × kf × MAX_POS), MAX_POS − current_pos))
MAX_POS25 units per contract
EDGE_MIN0.04 — minimum divergence between our_p and mkt_p to trade
Half-Kelly (0.5×)Conservative: protects against model error. Full Kelly would be 2× more aggressive.

Example: our_p=0.65, mkt_p=0.45, pos=0. kf=(0.65-0.45)/(1-0.45)=0.364. size=floor(0.5×0.364×25)=4 units.

CPI probability update How a CPI print changes our Fed probability estimates
signal
▶
surprise = actual − forecast
shift = CPI_SENS × |surprise| (CPI_SENS = 80)
surprise > 0 (hawkish)R_CUT probability -= shift; R_HIKE += shift
surprise < 0 (dovish)R_HIKE probability -= shift; R_CUT += shift

signal

Logic: if dovish and not hawkish → dovish shift. If hawkish and not dovish → hawkish shift. If neutral → neutral shift. If mixed or unclassified → logged but no probability change.

On competition day: verify the actual message_type string from news.log during practice. It must match one of FEDSPEAK_MSG_TYPES or it will be logged as "UNKNOWN TYPE" and not processed.

Settlement trade End-of-round concentrated bet when conviction is high and EV is positive
new alphafires once/round
▶

Fires once per round in the T-90s to T-20s window (70-second opportunity).

Three conditions must all be true:

  1. our_p ≥ 0.78 for the leading contract
  2. Positive EV: ask < 1000 × our_p (expected payout exceeds cost)
  3. Position room: current pos < SETTLE_MAX_POS = 35
SETTLE_MIN_P0.78 — probability threshold
SETTLE_MAX_POS35 units (larger than normal 25-unit cap)
Quantitymin(10, room) where room = 35 − current_pos
Order typeplace_order() directly (bypasses 25-unit safe_place_order cap)
RationaleNear round end, all CPI prints accumulated → most accurate probability estimate → largest justified bet
Trade cadence 15-second rate limit in main loop; bypassed on news events
timing
▶

The main loop calls strat_fed.trade() at most once every 15 seconds. This prevents 9,000 order attempts per round (old behaviour at 0.1s cadence).

However, bot_handle_news calls await strat_fed.trade() immediately after CPI and fedspeak. This preserves all news alpha — we trade the moment a signal arrives.

trade() always: (1) reads book probs, (2) cancels all existing Fed quotes, (3) places new Kelly-sized orders. The cancel-replace happens every trade call.

Options Strategy

OptionsStrategy class — arb checks + implied B price

Why options arb rarely fires Mathematical proof that friction exceeds available edge
structural
▶

Box spread (4 legs):

Theoretical value = K2 − K1 = 50 (for adjacent strikes)
Buy friction = C1_ask − C2_bid + P2_ask − P1_bid ≈ 4 × 3 = 12 ticks
Buy edge = 50 − (50 + 12) = −12 ✗ never profitable

PCP arbitrage (3 legs):

3-leg friction ≈ 3 × 3 = 9 ticks
Exchange option bots maintain PCP within 1-2 ticks
Available mispricing ≈ 2 ticks < 9 ticks friction ✗

We still run these checks every 1.5 seconds with MIN_EDGE=2. On the rare occasion an exchange bot makes a pricing error, we would capture it.

No-cancel-before-recheck (fix) Resting arb orders should stay in the book, not be cancelled every 1.5 seconds
fixed in final bot
▶

Old strategy.py cancelled all B and option orders before every arb check (every 1.5s). This caused two problems:

  1. Profitable orders cancelled: A resting arb order at a good price is a lottery ticket with positive EV. Cancelling it throws it away.
  2. Vol-quota race: The exchange still counts being-cancelled orders against the outstanding volume limit for a brief period. Immediately placing new orders then hit the limit and were rejected.

Fix: remove the cancellation loop entirely. The _cold() cooldown (4 seconds per arb type) prevents firing the same arb too often. Resting orders stay in the book until they fill or the round ends.

critical path

1. Record old_eps. Update eps["C"] = eps. 2. If PE_C_scale is None: set PE_C_scale = C_mid / eps 3. If calibrated: adjust anchor_mid += (eps − old_eps) × PE_C_scale 4. Call recalc_fv() → update fair_values["C"] 5. If FV is available: cancel C quotes and sweep stale book orders
Fedspeak headline Keyword parse → probability update → immediate Fed trade ▶
1. strat_fed.on_fedspeak(content) # hawkish / dovish / neutral shift 2. strat_c.recalc_fv() # update C FV with new ERC 3. await strat_fed.trade() # immediate Kelly trade

The 8 Changes

Everything changed from strategy.py to final_bot.py, with exact reasoning

Change 1 — Fed position bug (CRITICAL) max(1,...)→max(0,...) + max_abs_position=25 + 15s cadence
was causing -150k rounds
▶

Bug: _kelly_size returned max(1, ...), always returning at least 1 even when pos=MAX_POS=25. safe_place_order used max_abs_position=200 (not 25). Every 0.1s loop iteration placed another order. Fed accumulated to ±200.

Fix 1: Change to max(0, min(target, MAX_POS - pos)).

Fix 2: In __init__, max_abs_position["R_HIKE"] = max_abs_position["R_HOLD"] = max_abs_position["R_CUT"] = 25.

Fix 3: Call strat_fed.trade() every 15s in main loop (not every 0.1s). Still fires immediately on CPI/fedspeak.

Impact: Worst-case Fed loss: 25×1000 = -25k (was -200k). Eliminates rank-42 rounds from Fed blowup.

Change 2 — EPS_SCALE_A auto-calibration Self-corrects scale from market observations. Median of 3+ events.
from v3, kept
▶

On each A earnings: record implied = mid / (eps × PE_A). After 3+ observations, update EPS_SCALE_A to the median if it diverges by more than 1.0. _scale_obs persists across rounds.

Why needed: If EPS_SCALE_A is wrong, the first sweep goes in the completely wrong direction, generating large losses before the model self-corrects.

Change 3 — Remove A cross-spread at limit At pos=±200, suppress accumulating side instead of crossing spread
alpha preservation
▶

Old code: if pos >= MAX_ABS_POS: ask = best_bid — set ask price equal to the bid, crossing the spread to force a fill. This cost the full spread as a loss.

New code: if pos >= MAX_ABS_POS: bid = 0 — suppress the accumulating side. The reducing side (ask) continues to quote normally, attracting natural contra-flow without realising a spread loss.

Also fixed: Cross-book check now unconditional (previously gated on abs(pos) < MAX_ABS_POS, which was the condition that allowed the crossed-spread order to actually execute).

Change 4 — Stock C model replacement Gradient descent → empirical ERC sensitivity. max_abs_position=50.
largest new alpha source
▶

Why gradient descent failed: ERC barely varies in early minutes → gradients ≈ 0 → BETA/GAMMA never converge → C FV is garbage → runaway accumulation in wrong direction.

New model: C_FV = anchor_mid + sensitivity × (current_ERC − anchor_ERC). Single parameter calibrated from one CPI observation. Valid immediately.

New alpha: Directional C trades on every CPI post-calibration (±10 units). Expected +5-12k/round that was previously zero.

Also fixed: C FV reset to None at round start. max_abs_position["C"] = 50.

Change 5 — Options cancel-before-recheck removed Resting arb orders stay in book. Removed vol-quota race.
correctness fix
▶

Removed the loop in OptionsStrategy.run() that cancelled all B and option orders every 1.5s before re-checking arb. Two reasons this was wrong: (1) resting arb orders are profitable when they fill; (2) cancellation + immediate re-placement causes a brief period where the being-cancelled order still counts against the outstanding volume limit.

Change 6 — Fed settlement trade End-of-round bet when our_p ≥ 0.78 and EV is positive
new alpha from v3
▶

New method FedStrategy.settlement_trade(). Fires once per round in T-90s to T-20s window. Requires (1) leading contract's our_p ≥ 0.78, (2) ask < 1000 × our_p (positive EV), (3) position room below SETTLE_MAX_POS=35.

Uses place_order() directly to allow positions up to 35 (vs normal safe_place_order cap of 25). The higher cap is appropriate only for this end-of-round terminal bet.

Change 7 — ETF: implied B + one-sided cooldown PCP-implied B in NAV. 3-second rate limit on one-sided trades.
precision + safety
▶

NAV improvement: ETFStrategy now holds a BPriceTracker. _nav() uses PCP-implied B price (when available) instead of B book mid. More accurate NAV → fewer false arb signals, more true ones captured.

Rate limit: One-sided trades now capped at once per 3 seconds. Old behaviour fired every 0.5s when threshold was met, which could place 6 orders per second consuming outstanding vol quota entirely.

Change 8 — Main loop cleanup Remove record_obs, add settlement window check, reset C FV at round start
hygiene
▶
  • Removed strat_c.record_obs() call — no gradient descent anymore.
  • Added last_fed_trade variable and 15-second gate in the loop.
  • Added settlement trade check: if ROUND_DURATION_SECS - 90 ≤ elapsed < ROUND_DURATION_SECS - FLATTEN_LEAD_SECS.
  • Added self.fair_values["C"] = None in _on_round_start().
  • Added last_fed_trade = now when round resets.

What Not To Do

Things that look like improvements but make the bot worse

SNIPE_MAX_PENDING — do not add this Caps pending A snipe orders. Throttles the sweep. Reduces primary alpha.
known regression
▶

v3 added a cap of 2 pending snipe orders for A. The intent was to prevent order pile-up. The effect was to limit how many stale book levels we could sweep in the 2-second post-earnings window. Strategy.py has no such cap and ranks higher because it sweeps more aggressively. Do not add this.

Stale order cleanup — do not add this Cancels orders unfilled for >20s. But unfilled orders may be profitable resting orders.
known regression
▶

v3 added a function that cancelled any limit order unfilled for more than 20 seconds. The logic was to free up order slots. The effect was to cancel profitable resting orders that were waiting for the market to return to their price. A limit order resting in the book at a profitable price is a lottery ticket with positive EV. Do not cancel it.

FlowTracker.ALPHA = 0 — do not disable this v3 disabled flow tracking citing "death spiral risk". Risk is now handled by auto-calibration.
known regression
▶

The death spiral risk was: wrong FV → sell A → flow sees sell pressure → fv_adj drops → more selling. This risk disappears once EPS_SCALE_A is correctly calibrated (which auto-calibration ensures by round 1, event 3). With correct scale, flow is a genuine edge that adjusts FV toward where smart money is trading. Keep ALPHA=0.05.

FV_CLAMP_PCT — do not add this v3 clamped A FV to ±12% of book mid. Unnecessary and limits earnings alpha.
known regression
▶

The clamp was v3's response to the problem of wrong EPS_SCALE_A producing a wildly incorrect FV. With auto-calibration (Change 2), the scale corrects within 3 events. The clamp is unnecessary and costs alpha when a large legitimate EPS move exceeds 12% — the clamp would have prevented the full sweep of that move.

Heavy PnL logging — do not add more logging Every file write in the hot path adds latency. Latency costs sweep fills.
latency risk
▶

v3 added detailed PnL attribution logging to full_pnl.log, fed_pnl.log, a_pnl.log, etc. These file writes happen in the main event handling path. In a market where the post-earnings window is 2 seconds and book updates fire 30-50 times per second, extra file I/O adds measurable latency. Keep logging minimal: bot.log (console tee), fills.log, fed.log, news.log, pnl.log (status only), meta.log.

Aggressive directional C before calibration — do not add this Trading C without a calibrated sensitivity is random directional betting.
model risk
▶

Before the ERC sensitivity is calibrated, we have no reliable FV for C. Trading C in this phase (as v1-v3 did, via the gradient descent model that was giving garbage BETA/GAMMA) built up large positions in wrong directions. The three-phase lifecycle is the correct design: trade C only after calibration. Be patient.
